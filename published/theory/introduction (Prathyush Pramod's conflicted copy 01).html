1 2.1 Introduction

The preliminary conception of this idea came into shape when I started using systems enlisted in my university to take care of digital side of student academics. I started using these applications at a time when I was being exposed to the core principles of Computing Science and I felt a major discord on contrasting architecture of apps deployed in an academic setting with principles that I learnt in Computing Science.

Also, at this time, I started building a few apps in my extra time to help my colleagues but overtime I found out that my own apps suffered from the same kind of systemic errors found in the academic environment.

And since then I have been building smaller systems that addressed these problems and I see this dissertation as an attempt at unifying all those under a single umbrella and uniting them into a single platform.

It made me realize that much better can be done by evolving software systems when you figure out a good architectural fit foremost.

Hence, the topic of this dissertation is set as an attempt at unifying the efforts that I have done over my 3 year of study and also in order to rectify the problems that I have been observing when creating a user facing software.

Even though it might look like the architecture was figured out foremost and the apps were shoehorned to fit it, the truth is quite the opposite. This architecture was formed mostly in hindsight after a lot of experimentation and constantly evolving it over time to accomodate the corner cases. Even, the architecture as it currently stands is not to be held as absolute and constant pruning and evolution is required to adapt it to newer content.

The process was build, observe, evaluate, rebuild. Similar to the OOPA loop suggested by John Boyd. This loop is to be kept in place if the architecture evolves to encompass the potentially new systems.

It was essential to create distinct visual styles for distinct apps and probably I have tried to explore different sets of architectural styles for each apps. For example the ExamVault website follows a typical client server model with most of the visual rendering predetermined on the server side whereas in NCLProgress the majority is done on the client side, similarily for IvoryTower which forms the backbone of apps by providing them data uses a RESTful architecture in delivering the information. I will be detailing about these styles later in fuller detail later in the thesis but it is to be noted that modularizing the apps and creating a domain for a coherent set of functionality enables the exploration of creating bespoke architectures that are well fitted to each domain. This could be a major win from a polyglot perspective as I have mentioned this enables for integrating newer systems into the ecosystem with ease.

“Understanding the relationship between web technology and the underlying Internet protocols is fundamental to the design and implementation of true web applications. In fact, it is the exploitation of that relationship that distinguishes a web page or web site from a web application.”

Excerpt From: Rosen, Rich. “Web Application Architecture: Principles, Protocols and Practices.”

Systems had to be constructed with the idea that specifications would always change. And as the saying goes change being the only thing that is constant, it was high priority to build this ecosystem in a manner that it accomodated change. This has been achieved by keeping the systems modular.
